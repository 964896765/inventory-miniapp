# 项目约束文档

> 本文档定义了项目开发过程中必须遵守的硬性约束和规则
> 任何阶段的开发都不得违反这些约束

---

## 1. UI约束

### 1.1 首页（Home Tab）

**约束级别**：🔴 严格禁止修改

**规则**：
- ✅ 首页UI布局不得改动
- ✅ 所有入口和卡片结构必须保持现状
- ✅ 只允许补齐数据展示与跳转逻辑
- ❌ 禁止调整控件位置
- ❌ 禁止修改布局结构
- ❌ 禁止删除或新增卡片区域

**允许的操作**：
- 补充数据展示（如统计数字、列表数据）
- 完善跳转逻辑（如点击卡片跳转到对应页面）
- 优化数据加载逻辑

**验收标准**：
- 首页截图与初始状态对比，布局完全一致
- 所有控件位置不变
- 卡片数量和排列顺序不变

### 1.2 工作台（Workspace Tab）

**约束级别**：🟡 必须重构

**强制要求**：
- ✅ 顶部必须为五大仓库横向Tabs
- ✅ Tabs必须支持吸顶（sticky）效果
- ✅ 中部必须为左右双栏联动布局
- ✅ 底部必须有两个固定按钮
- ❌ 不允许使用其他布局方式

**布局规范**：
```
┌─────────────────────────────────────┐
│  主材仓 | 车间仓 | PACK | 辅料 | 待处理  │ ← 吸顶Tabs
├──────────┬──────────────────────────┤
│          │                          │
│  左栏    │       右栏               │
│  (分类/  │     (材料列表)           │
│   部门)  │                          │
│          │                          │
├──────────┴──────────────────────────┤
│  [按BOM发料]  [退仓/换料/平账]       │ ← 固定底部
└─────────────────────────────────────┘
```

**数据联动规则**：
- 主材仓：左栏=材料分类，右栏=该分类材料列表
- 车间仓：左栏=部门列表，右栏=该部门结存材料列表
- PACK/辅料/待处理：左栏=材料分类，右栏=该分类材料列表

### 1.3 我的（Profile Tab）

**约束级别**：🟢 允许扩展

**必须包含的入口**：
- ✅ 个人信息
- ✅ 成员管理

## 3. 数据库约束

### 3.1 五大仓库

**约束级别**：🔴 严格要求

**默认仓库**（必须存在）：
1. 主材仓
2. 车间仓
3. PACK仓
4. 辅料仓
5. 待处理

**规则**：
- ✅ 必须在系统启动时自动创建（幂等）
- ✅ 顺序必须固定（使用sort字段）
- ✅ 不得删除这五个仓库
- ✅ 可以修改名称，但sort顺序不变

**实现要求**：
- 添加 `warehouses.sort` 字段
- 创建 `warehouses.ensureDefaults()` 方法
- 服务启动时自动执行

### 3.2 部门来源

**约束级别**：🔴 严格要求

**规则**：
- ✅ 部门必须来自 `supply_units` 表
- ✅ 部门类型为 `type='department'`
- ✅ 不得创建独立的部门表
- ✅ 部门与车间仓子仓库一一对应

**数据一致性**：
- 创建部门时，自动创建对应的车间仓子仓库
- 删除部门时，同步删除对应的子仓库（需检查是否有库存）
- 部门名称修改时，同步更新子仓库名称

### 3.3 库存三层架构

**约束级别**：🔴 严格要求

**强制架构**：
```
docs (单据主表)
    ↓
doc_items (单据明细)
    ↓
stock_ledger (库存流水)
    ↓
inventory (库存汇总)
```

**规则**：
- ✅ 任何库存变动必须走单据
- ✅ 单据必须生成流水
- ✅ 流水驱动库存更新
- ❌ 禁止直接修改inventory表
- ❌ 禁止跳过单据直接操作流水

## 4. 业务规则约束

### 4.1 车间仓部门结存规则（最重要）

**约束级别**：🔴 最高优先级

**核心规则**：
> 凡是出库到"车间仓"的（无论是按BOM发料，还是直接出库）都必须计入车间仓对应部门的结存。

**详细说明**：
1. **出库到车间仓的流程**：
   - 用户选择目标仓库=车间仓
   - 系统强制要求选择部门
   - 执行出库：主材仓OUT + 部门仓IN
   - 记录两条流水：主材仓减少、部门仓增加

2. **BOM发料的流程**：
   - 选择BOM编号和部门
   - 加载BOM材料清单
   - 执行发料=出库到车间仓部门仓
   - 主材仓减少，部门仓增加

3. **禁止的操作**：
   - ❌ 不允许只减少主材仓而不增加部门结存
   - ❌ 不允许出库到车间仓时不选择部门
   - ❌ 不允许BOM发料时跳过部门结存

**验收标准**：
- 出库到车间仓后，部门结存必须增加
- BOM发料后，部门结存必须增加
- 工作台车间仓Tab右侧能看到部门结存数量变化
- 材料流水页能看到对应的流水记录

### 4.2 部门仓实现方案

**约束级别**：🔴 强制使用方案A

**方案A（推荐并强制）**：部门 = 车间仓子仓库
```
warehouses表结构：
- 车间仓（parentId=NULL）
  - 配料部门仓（parentId=车间仓Id, departmentId=配料部门Id）
  - 制片部门仓（parentId=车间仓Id, departmentId=制片部门Id）
  - 卷绕部门仓（parentId=车间仓Id, departmentId=卷绕部门Id）
  - ...
```

**禁止方案B**：inventory增加departmentId字段
- ❌ 不允许使用此方案（改动大，不推荐）

**实现要求**：
- 添加 `warehouses.departmentId` 字段（可选）
- 创建 `warehouseDepartments.sync()` 方法
- 将 `supply_units.department` 同步为车间仓子仓库（幂等）

### 4.3 审批流程规则

**约束级别**：🔴 严格要求

**单据状态流转**：
```
draft (草稿)
    ↓ submit()
submitted (已提交)
    ↓ approve()/reject()
approved/rejected (已审批/已驳回)
    ↓ post() (仅approved可过账)
posted (已过账)
```

**审批配置规则**：
- 是否启用审批
- 审批级别（一级/二级）
- 免审规则（审批人自己提交的可免审）
- 审批人配置（一级审批人/二级审批人）

**审批记录规则**：
- 每一次审批动作必须写入 `approval_logs`
- 记录内容：docId、审批人、动作（通过/驳回）、原因、时间、级别
- 驳回必须填写原因

## 9. 约束优先级

### 优先级定义

- 🔴 **严格禁止/严格要求**：必须100%遵守，违反即失败
- 🟡 **必须实现**：必须完成，但实现方式可灵活调整
- 🟢 **建议遵守**：推荐遵守，但不强制

### 冲突处理

**当约束之间存在冲突时**：
1. 🔴 优先级最高，必须优先满足
2. 🟡 次之，在不违反🔴的前提下满足
3. 🟢 最后，在满足🔴和🟡的前提下尽量满足
